# Story 21.7: Core Frontend Service Unit Tests

Status: done

## Story

As a developer,
I want unit test coverage for the load-bearing HTTP plumbing in the frontend (`api.service.ts` ‚Äî the NSwag-generated `ApiClient` ‚Äî and `auth.interceptor.ts` ‚Äî the functional `HttpInterceptorFn`),
so that token injection, 401 ‚Üí refresh ‚Üí retry orchestration, and the request-shape contracts the entire app depends on are pinned with regression tests instead of being exercised only end-to-end.

## Acceptance Criteria

> **Reality check (epic vs. shipped code):** Epic 21's text for Story 21.7 describes `api.service.ts` as "the base HTTP service used by all feature services" with hand-written `get/post/put/delete` methods and a `/api/v1` prefix that the service applies. The shipped code is different: `frontend/src/app/core/api/api.service.ts` is a **328 KB NSwag-generated `ApiClient`** (auto-generated from the swagger doc, regenerated by `npm run generate-api`). The `/api/v1/...` prefix lives inside each generated method (e.g., `auth_Login` ‚Üí `this.baseUrl + "/api/v1/auth/login"`). The injectable base URL comes from the `API_BASE_URL` injection token (`frontend/src/app/core/api/api-base-url.token.ts`, defaulting to `""` per `app.config.ts` so requests fall through the dev proxy or hit the same origin in prod). The ACs below test **the shipped behavior**:
>
> - URL composition (verifies the generated client respects the `/api/v1/...` route and the `API_BASE_URL` token).
> - Method/body/withCredentials/Accept/Content-Type header contracts on a representative slice of methods (auth + properties + expenses).
> - Error propagation through the generated `processX` pipeline so callers see real `HttpResponseBase`-derived errors with status preserved.
>
> **Do NOT** edit `api.service.ts` ‚Äî it is auto-generated and any manual edit will be wiped on the next `npm run generate-api`. The single allowed manual edit (the `API_BASE_URL` import swap) is automated by `frontend/scripts/patch-api-service.js`.
>
> **Authorization flow under test (the auth interceptor):** `frontend/src/app/core/auth/auth.interceptor.ts` is a **functional** `HttpInterceptorFn`. It:
> 1. Adds `Authorization: Bearer <token>` when `authService.accessToken()` is non-null AND the URL is NOT `/api/v1/auth/...`
> 2. On a 401 from a non-auth-endpoint, calls `authService.refreshToken()` once, then retries the original request with the new token
> 3. If a refresh is already in flight (`authService.isRefreshingToken === true`), waits on `authService.refreshTokenInProgress` (a `BehaviorSubject<boolean>`) until it emits `false`, then retries with the freshly-set token
> 4. On refresh failure, calls `authService.logout()` and `router.navigate(['/login'])`
> 5. Does NOT trigger refresh on 401 from auth endpoints themselves (login/refresh/logout return 401s as the caller's problem, not the interceptor's)
>
> All tests use Vitest as the runner (project convention ‚Äî `ng test` / `npm test` runs Vitest, NOT Jasmine), `TestBed.configureTestingModule` for DI, `provideHttpClient()` + `provideHttpClientTesting()` (Angular 21 standalone HTTP testing API ‚Äî NOT the deprecated `HttpClientTestingModule`), and `HttpTestingController.expectOne(...)` / `flush(...)` / `verify()` for assertions.

## Dev Notes

### Test Scope

This is a pure frontend unit-test story. The deliverable IS the unit tests.

| Layer | Required? | Justification |
|---|---|---|
| **Unit (Vitest)** | **Required ‚Äî this IS the story** | `api.service.ts` (the NSwag `ApiClient`) and `auth.interceptor.ts` are load-bearing infrastructure used by every feature in the app and currently have **zero** dedicated unit tests. The `auth.service.spec.ts` file exists and covers the auth service itself, but the interceptor's behavior and the generated client's request-shape contracts are unverified at the unit level. |
| **Integration (.NET WebApplicationFactory)** | Not required | No backend changes. The interceptor's 401/refresh behavior is only meaningful client-side ‚Äî the corresponding refresh endpoint is already covered by integration tests for `AuthController`. |
| **E2E (Playwright)** | Not required | No new endpoints, components, routes, or user-facing flows. The existing E2E suites that exercise login/logout/protected-route navigation already implicitly cover the interceptor through real HTTP. This story is about pinning the unit-level contract, not adding user-level coverage. |

### Tech-stack reality (verified ‚Äî Apr 2026)

- Angular **21.2** (per `frontend/package.json`). The story title in the epic mentions "Angular 20+" ‚Äî the shipped code is on 21.2 and uses the same standalone HTTP testing API the docs describe for 20.x. No code-path change.
- HTTP testing API: **`provideHttpClient()` + `provideHttpClientTesting()`** registered in `TestBed.providers` (in that order ‚Äî `provideHttpClient()` MUST come first per the Angular docs, or `provideHttpClientTesting()` will not overwrite the right pieces of the configuration). The legacy `HttpClientTestingModule` is deprecated; do not use it. (Verified via Ref MCP against `https://angular.dev/guide/http/testing`, Apr 2026.)
- Interceptor API: **functional `HttpInterceptorFn`** registered via `provideHttpClient(withInterceptors([authInterceptor]))`. The class-based `HTTP_INTERCEPTORS` token + `withInterceptorsFromDi()` path is NOT used in this codebase ‚Äî do not test against it. (Verified via `frontend/src/app/app.config.ts:15`.)
- Test runner: **Vitest** (per `frontend/vitest.config.ts`). Use `import { describe, it, expect, beforeEach, afterEach, vi } from 'vitest'`, NOT Jasmine globals. `vi.fn()` for spies, `vi.mocked(x)` for typing, `vi.restoreAllMocks()` if needed. The existing `auth.service.spec.ts` and `auth.guard.spec.ts` are the canonical pattern.
- Run command: `npm test` (delegates to `ng test` which runs Vitest). Do NOT invoke `npx vitest` directly per project memory note.

### What `api.service.ts` actually is (vs what the epic says)

The epic's description of `api.service.ts` as a hand-written base service with `get/post/put/delete` methods that prefix `/api/v1` is **inaccurate for the shipped code**. The actual file is the NSwag-generated `ApiClient`:

- **Path:** `frontend/src/app/core/api/api.service.ts` (~328 KB, regenerated by `npm run generate-api`)
- **Companion:** `frontend/src/app/core/api/api-base-url.token.ts` (declares the `API_BASE_URL` `InjectionToken<string>` so the token can be tree-shaken from `app.config.ts` without pulling in the full client)
- **Patch script:** `frontend/scripts/patch-api-service.js` runs after NSwag and rewrites the inline `API_BASE_URL` declaration to import from the companion file. **Don't bypass this** ‚Äî manual edits to `api.service.ts` will be overwritten on the next regeneration.
- **Method shape (verified by reading lines 182-244 of the file for `auth_Login`):**
  - URL composition: `let url_ = this.baseUrl + "/api/v1/<route>";`
  - Body: `JSON.stringify(request)` then passed as `body: content_` to `this.http.request("post", url_, options_)`
  - Headers: explicit `Content-Type: application/json` (when there's a body) and `Accept: application/json`
  - `withCredentials: true` on every method (verified across auth methods; non-auth methods may differ ‚Äî check the source for the specific method under test)
  - `responseType: "blob"` ‚Äî the response body is converted via `blobToText` inside the `processX` pipeline, then JSON.parsed with the optional `jsonParseReviver`
  - Error path: any non-200 (or non-204 for void methods) status throws via `throwException("...message...", status, _responseText, _headers, result)` which produces a thrown `{ message, status, response, headers, result }` shape

### What the interceptor actually does (the 5-branch decision tree)

Read `frontend/src/app/core/auth/auth.interceptor.ts` end-to-end before writing tests. Summary:

```
+----------------------------------------------+
| 1. token AND not auth endpoint ‚Üí add header  |
| 2. otherwise ‚Üí leave request unchanged       |
+----------------------------------------------+
                  |
                  v
              [next(req)]
                  |
        +---------+----------+
        |                    |
     success                401
        v                    |
  pass through      isAuthEndpoint?
                       |        |
                      yes       no
                       |        |
              throw 401     isRefreshingToken?
                              |        |
                            true       false
                              |        |
                       wait on $        refresh()
                              |        |
                       got new token?   succeeded?
                          |    |          |    |
                        yes   no         yes   no
                          |    |          |    |
                       retry  /login   retry   logout()+/login
```

The two "wait on `refreshTokenInProgress`" tests (AC-15, AC-16) are the trickiest ‚Äî they require simulating the BehaviorSubject mid-test. The auth.service.spec.ts already establishes the pattern of stubbing the BehaviorSubject; reuse it.

### Pattern reference ‚Äî `auth.service.spec.ts`

`frontend/src/app/core/services/auth.service.spec.ts` is the **primary pattern reference** for the api.service.spec.ts. Verified-good practices to mirror:

- Imports from vitest (`describe, it, expect, beforeEach, afterEach, vi`)
- `TestBed.configureTestingModule` with `provideHttpClient(), provideHttpClientTesting()`
- `httpMock = TestBed.inject(HttpTestingController)`
- `afterEach(() => { httpMock.verify(); vi.restoreAllMocks(); })`
- `expectOne('/path').flush(body)` style assertions
- Use of `let response: T | undefined; service.foo().subscribe(r => response = r); httpMock.expectOne(...).flush(...); expect(response).toEqual(...)` for synchronous-feeling assertions (Vitest fake-timers not required)

Pattern reference for the interceptor spec ‚Äî `auth.guard.spec.ts`:

- `signal<T>()` + `Object.defineProperty`-style stub of services that expose signals
- `vi.fn().mockResolvedValue(...)` / `mockReturnValue(of(...))` / `mockReturnValue(throwError(() => ...))`
- `TestBed.runInInjectionContext` IS NOT NEEDED for the interceptor tests ‚Äî `httpClient.get(...)` triggers the interceptor through DI naturally; the guard tests use `runInInjectionContext` only because guards are functions that themselves call `inject()` outside of a normal injection scope.

### Anti-pitfalls

1. **Don't manually edit `api.service.ts`** ‚Äî it's auto-generated. The spec file lives next to it, but only `*.spec.ts` and the patch script touch this directory.
2. **Don't use `HttpClientTestingModule`** ‚Äî it's deprecated. Use `provideHttpClientTesting()`.
3. **Don't reverse `provideHttpClient()` and `provideHttpClientTesting()`** ‚Äî order matters; the Angular docs explicitly call this out as a footgun.
4. **Don't call `npx vitest` directly** ‚Äî run `npm test` from `/frontend`. (Project memory rule.)
5. **Don't assert against `req.request.body` as a parsed object** for NSwag methods ‚Äî the generated client calls `JSON.stringify(request)` itself, so the captured body is a JSON string. Either parse it or assert against the string.
6. **Don't expect `req.flush(blob)` to always work for blob-response methods** ‚Äî the testing backend's blob handling can be quirky. If a test fails because the parsed value isn't right, fall back to `req.flush(parsedObject)` and let the testing backend coerce. Document the chosen pattern in the test comment for future maintainers.
7. **Don't forget `httpTesting.verify()` in `afterEach`** ‚Äî without it, an interceptor test that fires a retry but never matches it will silently pass while leaving a pending request.
8. **Don't mock `Router` with bare `{}`** ‚Äî provide `navigate: vi.fn().mockResolvedValue(true)` so the interceptor's `router.navigate(['/login'])` returns a Promise (matches the real `Router.navigate` signature).
9. **Don't try to read the `Authorization` header from the original (un-cloned) request** ‚Äî the interceptor calls `req.clone({ setHeaders: { Authorization: ... } })`, so always assert against the matched request from `httpTesting.expectOne(...)`, which sees the cloned version.
10. **Don't write a test for the guard-redirect-on-init-failure path** ‚Äî that's `auth.guard.spec.ts`'s territory. This story is about the interceptor, not the guard.
11. **Watch for `firstValueFrom` on `Observable<void>`** (AC-7 / `expenses_DeleteExpense`) ‚Äî it rejects with `EmptyError` if the observable completes without emitting. Subscribe with `{ next, error, complete }` instead, OR use `lastValueFrom(observable, { defaultValue: undefined })`.
